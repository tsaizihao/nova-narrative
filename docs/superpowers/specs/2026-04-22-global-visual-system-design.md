# 叙世者全局视觉系统设计

## 文档目标

本文档定义 `叙世者` 当前桌面应用的全局视觉系统，目标不是单独修一个页面，而是为整个 APP 建立一套可复用、可扩展、可约束的界面语言。

本文档回答以下问题：

- 这款产品在视觉上到底应该成为什么，而不只是“某个页面好不好看”
- 为什么当前页面会反复陷入“浅色大圆角表单拼盘”
- 全局应该采用什么样的产品气质、层级规则和组件边界
- 导入、构建、审阅、游玩、结局五个阶段如何共享同一套设计系统
- 现有页面的改造优先级应该如何安排

本文档不是实现计划，也不是临时灵感记录。它是后续视觉统一、组件抽象和页面重构的设计依据。

## 设计结论

`叙世者` 的全局视觉方向定义为：

- `文档档案感 70%`
- `现代编辑器感 30%`

一句话定义：

`叙世者` 的界面应该像一套被整理过的叙事档案工作台，而不是后台管理系统，也不是被奶油色和大圆角覆盖的表单应用。

这意味着：

- 内容比控件更重要
- 标题比装饰更重要
- 留白比边框更重要
- 结构比“高级感皮肤”更重要

## 当前系统性问题

当前界面已经出现了暖底色、书卷标题和绿色主动作的雏形，但还没有被提炼成统一系统，因此页面会不断滑向以下问题：

- 页面、面板、输入框、按钮都在使用近似的浅暖底色，导致信息层次塌陷
- 多个层级的容器都使用大圆角，造成“页面像卡片，卡片像输入框，输入框像按钮”的混乱
- 很多页面将元信息、正文、状态、操作揉在一起，导致用户需要先看懂组件，再开始任务
- 主按钮经常依靠夸张的体积和胶囊圆角强调自己，而不是依靠层级和位置
- 表单字段、档案内容和辅助标签没有建立清晰的视觉语法，导致内容始终像在“填后台”

最典型的失败模式是：

`用户本来应该感觉自己在编辑故事档案，但最终却感觉自己在操作一套暖色 CMS 表单。`

## 产品视觉定位

全局产品定位应固定为：

`叙事档案工作台`

这一定义对五个阶段都成立：

- 导入阶段像把原稿铺到工作台上
- 构建阶段像系统正在整理叙事索引
- 审阅阶段像编辑和校正故事档案
- 游玩阶段像翻阅一份被激活的叙事材料
- 结局阶段像把本轮故事结果归档

这套产品不是：

- SaaS 后台
- 普通表单应用
- UI 炫技阅读器
- 靠装饰表达文学感的伪文艺工具

## 全局设计原则

### 1. 每一屏只能有一个主角

每个页面必须先回答：

`当前阶段用户最重要的任务是什么`

然后让一个区域承担这个任务，其他区域退为辅助。

应用到五个阶段时：

- 导入页主角是 `小说正文导入`
- 构建页主角是 `生成进度与当前阶段`
- 审阅页主角是 `当前正在编辑的条目`
- 游玩页主角是 `故事正文`
- 结局页主角是 `结局与回顾`

### 2. 信息必须分层，不准混层

全局信息统一分为四层：

- `标题层`
  页面名、章节名、条目名、结局名
- `元信息层`
  类型、插槽、标签、状态、数量、更新时间
- `内容层`
  正文、设定、摘要、预览结果、回顾内容
- `动作层`
  保存、删除、进入故事、刷新预览、继续游玩

规则：

- 元信息不能漂在正文区底部
- 动作层不能抢标题层的戏
- 内容层不能被表单边框切得支离破碎

### 3. 视觉重心从控件转向内容

界面不应依赖以下方式制造设计感：

- 到处增加圆角
- 到处增加边框
- 到处增加浅暖底色
- 把按钮做得又大又圆

界面应依赖以下方式建立质量：

- 稳定的标题层级
- 清晰的留白节奏
- 节制的品牌色使用
- 合理的信息分区
- 可预期的组件边界

### 4. 组件只能承担单一职责

全局容器只分三级：

- `Shell`
  页面级承载层
- `Panel`
  模块级组织层
- `Field`
  输入级编辑层

规则：

- Shell 不能大量出现
- Panel 不应冒充页面
- Field 不应长得像卡片

### 5. 品牌色必须稀缺

全局主品牌色为深墨绿，但它只能用于：

- 唯一主按钮
- 当前激活态
- 正向状态
- 焦点和关键反馈

不能用于：

- 大面积背景
- 普通输入框底色
- 大量同时出现的标签
- 每个面板的强调色

## 全局设计 Token

### 色彩

全局只保留以下核心色彩角色：

```text
bg/app           #F5F1EA
bg/canvas        #FBF8F3
bg/panel         #FFFDFC
bg/subtle        #F7F2EA
bg/field         #FCF9F4

border/subtle    #E8DED2
border/default   #D8C9B7

text/strong      #2C241D
text/default     #4B4036
text/muted       #867666

accent/brand     #1F6A57
accent/soft      #E4F0EC

danger           #B2534A
warning          #A7742A
success          #1F6A57
```

使用规则：

- `bg/app` 用于应用最外层背景
- `bg/canvas` 用于页面工作区背景
- `bg/panel` 用于主面板和正文承载
- `bg/subtle` 用于次级信息块
- `bg/field` 用于输入字段和弱编辑槽
- `accent/brand` 只保留给主操作和激活态

### 字体

全局采用双字体策略：

- 标题字体：
  `"Iowan Old Style", "Songti SC", "Source Han Serif SC", serif`
- 界面字体：
  `"SF Pro Text", "PingFang SC", "Hiragino Sans GB", "Noto Sans SC", sans-serif`

使用规则：

- Serif 只给标题层和少量叙事性大标题
- Sans 负责界面、输入、按钮、说明、标签
- 不允许全员 Serif，也不允许全员粗字

### 字号

全局固定六级文字体系：

```text
display    36/44  semibold
title      28/34  semibold
section    20/28  medium
body       16/24  regular
meta       13/18  regular
micro      12/16  medium
```

规则：

- 一个页面最多出现一个 `display`
- 一个模块最多出现一个 `title`
- `meta` 负责说明，不许用加粗冒充层级

### 间距

全局统一采用 `8pt` 节奏：

```text
4  8  12  16  24  32  40  48  64
```

推荐映射：

- 标签与输入框：`8`
- 同组字段：`16`
- 分组与分组：`32`
- 面板内边距：`24` 或 `32`
- 页面主区内边距：`40`
- 页面区块之间：`40` 到 `48`

### 圆角

全局只保留四档圆角：

```text
r-sm   10
r-md   14
r-lg   18
r-xl   24
```

规则：

- `999px` 仅给胶囊标签和极少数轻量按钮
- 输入框与按钮默认使用 `r-md`
- 面板默认使用 `r-lg`
- 页面级 Shell 使用 `r-xl`

### 阴影

阴影不负责制造“高级感”，只负责区分层级：

```text
shadow/sm   0 2 8   rgba(44,36,29,0.04)
shadow/md   0 8 24  rgba(44,36,29,0.06)
shadow/lg   0 16 40 rgba(44,36,29,0.08)
```

规则：

- 大多数输入框不应使用阴影
- 主面板、抽屉、阅读舞台可使用轻阴影
- 阴影优先少用，边框优先弱用

## 全局组件规范

### 1. Workspace Topbar

职责：

- 说明当前工作对象
- 承载阶段信息
- 提供轻量元信息

规则：

- 顶栏始终只有一层
- 左侧显示品牌或当前对象
- 右侧显示阶段、状态或辅助导航
- 顶栏不应演变成仪表盘

### 2. Shell

用途：

- 页面级主承载容器
- 阅读舞台
- 核心编辑工作台

规则：

- 一个页面最多一个强 Shell
- 使用 `bg/canvas` 或 `bg/panel`
- 使用 `r-xl`
- 不得内部再无意义叠加多个大壳子

### 3. Panel

用途：

- 编辑器
- 预览区
- 说明区
- 侧栏抽屉

规则：

- Panel 只负责组织，不负责抢视觉中心
- 默认使用 `bg/panel`
- 次级 Panel 才使用 `bg/subtle`
- 不再通过重复大圆角和重阴影制造“层次”

### 4. Buttons

全局按钮只保留四类：

- `Primary`
- `Secondary`
- `Ghost`
- `Danger`

规则：

- 一个区域最多只允许一个 `Primary`
- `Primary` 使用品牌绿底和浅色文字
- `Secondary` 使用浅底和细边框
- `Ghost` 用于轻操作
- `Danger` 默认不用实心红按钮，优先采用浅边框或文本危险按钮

推荐尺寸：

```text
sm  h36  px12-14
md  h44  px16-18
lg  h48  px20
```

禁忌：

- 不要默认大药丸
- 不要两个同级主按钮并列
- 删除按钮不能比保存按钮更显眼

### 5. Inputs

用途：

- 承载短文本编辑
- 承载结构化字段编辑

规则：

- 输入框像编辑槽，不像卡片
- 背景使用 `bg/field`
- 边框使用 `border/default`
- 圆角使用 `r-md`
- 默认字号使用 `body`
- 焦点态使用品牌色边框或焦点环
- 主内容字段默认独占一行，避免与元信息字段做等宽并排
- 类型、状态、优先级、来源这类元信息字段应进入头部属性行或次级属性区，不应伪装成正文主字段

禁忌：

- 不要用输入框承载模块级视觉分量
- 不要让输入框比所属 Panel 还显眼

### 6. Textarea

Textarea 是内容型产品中的关键组件，应被定义为“小型写作区”。

规则：

- 使用更宽松的内边距和行高
- 正文编辑区优先使用 `17/30` 左右的阅读型排版
- 最小高度按语义分级，不统一做成超大方块

建议分级：

- `短摘要`：96 到 120
- `中段说明`：160 到 220
- `正文编辑`：260 以上

### 7. Select

规则：

- 视觉语法与 Input 保持一致
- 高度、圆角、边框必须对齐
- 箭头图标弱化
- 不得做成独立的奶油色大块
- 更适合出现在属性行、筛选行和状态区，不适合与“名称 / 标题 / 正文摘要”类主内容字段做 `1:1` 对排

### 8. Switch

Switch 承担的是状态职责，不是装饰职责。

规则：

- 永远用 `标签 + switch` 水平组合
- 文案应明确说明状态含义
- 开启态使用品牌绿
- 关闭态保持中性

禁忌：

- 不要做成左边一个标题、右边一个孤立勾选图标，中间空一大片

### 9. Chips

Chip 是全局元信息语法的一部分，用于表达：

- 类型
- 插槽
- 状态
- 数量
- 来源

分类：

- `Type chip`
- `Status chip`
- `Meta chip`

规则：

- Chip 默认不是按钮
- 只承载短文本
- 颜色对比必须克制
- 使用 `micro` 或 `meta` 尺寸

### 10. List Item

角色、世界书、规则、场景等列表项统一定义为“档案索引项”。

结构：

- 主标题
- 一行摘要或身份说明
- 可选状态标签

规则：

- 默认态像可选档案，而不是导航按钮
- 激活态通过边框和浅色底区分
- 不通过重色块整块高亮

### 11. Action Bar

编辑页统一引入底部动作栏语法。

职责：

- 聚合保存、删除、状态反馈
- 固定编辑闭环

结构建议：

```text
[状态提示]                     [删除条目] [保存更改]
```

规则：

- 危险动作靠左
- 主动作靠右
- 状态提示与动作同层管理
- 不把“未保存更改”丢到页面角落孤立显示

## 五个阶段的页面规则

### 1. 导入页

主任务：

`把原稿放进系统`

页面定义：

- 主区承载项目名称和正文导入
- 辅区承载导入提示、AI 设置、可继续项目
- 正文输入区是导入页唯一主角

规则：

- 正文区必须像稿纸输入区，而不是营销卡片
- AI Provider 是支持配置，不应抢正文的视觉位置
- “导入 txt / 载入示例 / 开始改编”必须形成清晰动作分组

### 2. 构建页

主任务：

`让用户理解系统正在整理什么`

页面定义：

- 单核心进度面板
- 标题说明当前构建阶段
- 进度与产出对象可见

规则：

- 视觉上像“编目中”，不是“空白等待页”
- 不新增复杂操作
- 进度条应稳，不应炫

### 3. 审阅页

主任务：

`编辑条目并即时验证`

页面定义：

- 左侧为索引或类型切换
- 中间为当前条目编辑器
- 右侧为预览或校验 rail

规则：

- 当前编辑条目是页面唯一主角
- 左侧列表是索引，不是主视觉区域
- 右侧预览是旁注和证据，不是第二主舞台
- 角色卡、世界书、规则三类编辑器必须共享同一套编辑语法

针对条目编辑页的直接要求：

- 条目名必须成为页面主标题
- 类型、插槽、状态必须上移到标题区
- 正文区必须成为视觉重心
- 保存和删除必须进入统一 Action Bar

### 4. 游玩页

主任务：

`沉浸阅读并做出回应`

页面定义：

- 顶部为极简章节信息和退出入口
- 中间为故事正文舞台
- 底部为响应 dock
- 两侧为世界设定与状态日志抽屉

规则：

- 正文始终最大、最安静、最可读
- 控件应服从阅读，不打断阅读
- 抽屉应像边注，不应像另一个后台页面

### 5. 结局页

主任务：

`收束、回顾、归档`

页面定义：

- 主标题为结局结论
- 中区展示关键回顾
- 辅区展示规则、角色、关键节点
- 底部承载重新开始或返回审阅等动作

规则：

- 语气比游玩页更静
- 更像结果归档页，而不是操作列表页
- 强调“你达成了哪个版本的故事”

## 页面之间的统一叙事

五个阶段共享的体验逻辑应是：

1. 导入：铺开原稿
2. 构建：整理档案
3. 审阅：编辑档案
4. 游玩：激活档案
5. 结局：归档结果

这才是全局统一性的来源。

统一性不等于所有页面长得一样，而是所有页面都属于同一个叙事工作流。

## 改造优先级

### 第一优先级：审阅条目编辑器

目标：

- 先消除“浅色大圆角表单拼盘”
- 建立档案头部、正文区和 Action Bar 语法

涉及方向：

- `CharacterReviewPanel`
- `WorldBookPanel`
- `RuleBookPanel`
- `ReviewWorkspace`

### 第二优先级：顶层工作台骨架

目标：

- 统一 `WorkspaceTopbar`
- 统一各阶段 Shell / Panel 层级
- 压缩重复的 hero 和说明区

涉及方向：

- `ImportScreen`
- `ReviewStageShell`
- `RuntimeStageShell`
- `BuildProgressScreen`

### 第三优先级：游玩与结局阶段收敛

目标：

- 进一步突出阅读舞台
- 弱化控件感
- 统一抽屉、状态块和结局回顾块语法

涉及方向：

- `ReaderDesktopShell`
- `ReaderMobileShell`
- `ReaderControlDock`
- `EndingScreen`

## 非目标

- 本次设计不重做产品信息架构
- 本次设计不引入新的后端能力
- 本次设计不新增复杂动效系统
- 本次设计不追求复古纸张拟物
- 本次设计不把应用做成深色主题优先产品

## 成功标准

当这套视觉系统落地后，用户进入任意一个阶段，都应感受到：

- 这是同一个产品，而不是多个页面各自设计
- 我在整理和阅读故事材料，而不是填写后台
- 页面主任务总是清楚的
- 状态、元信息、正文和动作各归其位
- 品牌感来自秩序和气质，而不是来自大量装饰
