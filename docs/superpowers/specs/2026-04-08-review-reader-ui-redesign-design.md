# 结构化故事引擎界面重构设计

## 背景
上一轮实现已经把结构化故事引擎 v1 的运行时能力接通，包括：
- 结构化 world model
- lore 激活与生命周期
- rule evaluation
- review phase
- 三栏 reader

功能已通，但当前界面仍有一个明显问题：
- `制作互动游戏` 和 `游玩互动游戏` 是两种不同任务
- 现有 UI 让它们共享了过于接近的卡片化表达
- 结果是 review 像后台表单堆叠，reader 也带有明显后台感

本次重构不改变运行时契约，也不改动核心数据模型；目标是把同一产品中的两类任务，分别放进最顺手的界面骨架里。

## 核心结论
产品保留单一流程，但采用双重心界面策略：

1. `import / build / review` 使用同一套 `工作台骨架`
2. `reader` 使用 `舞台骨架`
3. `reader` 的桌面端与移动端采用不同布局

这不是把产品拆成两个入口，而是在同一流程里明确切换任务心智：
- 前半程是在 `制作互动游戏`
- 后半程是在 `游玩互动游戏`

## 设计目标
### 目标
- 让首次使用者在每个阶段都知道“现在该做什么”
- 让 review 更像轻量编辑台，而不是后台表单墙
- 让 reader 更像视觉小说舞台，而不是结构化数据面板
- 保留结构化 lore / state / rules 的可见性，但不让它们压过故事正文
- 在桌面端与移动端都保持清晰和低学习成本

### 非目标
- 不新增新的故事运行时能力
- 不重做 review 的数据编辑范围，仍然只做轻量编辑
- 不引入完整自由建模工作台
- 不把 reader 改成纯图片导向视觉小说
- 不追求复杂视觉风格或强戏剧化装饰

## 总体原则
### 1. 单屏只允许一个主角
- `review` 的主角是当前编辑对象
- `reader` 的主角是当前故事场景

### 2. 信息要按任务优先级降级
- 主任务信息常驻
- 辅助信息保持“看一眼就懂”
- 系统性信息用摘要、抽屉、窄栏或胶囊表示，不与主内容同级争抢注意力

### 3. 同一产品允许不同骨架
只要导航与视觉语言连续，`制作` 和 `游玩` 不需要共享同一套版式。

### 4. 简单比“设计感”更重要
避免过度装饰、过多卡片层次、过多同权重区块。

## 信息架构
顶部保留统一的轻量阶段条：
- `导入`
- `构建`
- `审阅`
- `游玩`

阶段条只承担两个职责：
- 标明当前所处阶段
- 维持单流程产品的连续性

它不承担复杂导航，不展示大量次级操作。

## 工作台骨架
工作台骨架适用于：
- `import`
- `build`
- `review`

其核心结构是：
- 一个明确的主操作区
- 一个低干扰的辅助区
- 一个唯一主按钮或唯一主任务

### Import
布局要求：
- 主区为大文本输入区
- 页面只保留必要说明和一个主按钮
- 不放并列的说明卡片、能力卡片、示例卡片瀑布

用户看到页面时应该立刻知道：
- 这里是导入原文的地方
- 下一步就是开始生成

### Build
布局要求：
- 中央只显示当前进度、阶段名、简短说明
- 技术细节不作为默认信息出现
- 页面焦点是“系统正在把小说编译成可玩的故事”

用户看到页面时应该立刻知道：
- 当前进度到哪一步
- 还在等待，不需要额外决策

### Review
Review 必须像轻量编辑台，而不是整页表单合集。

布局采用 `主编辑区 + 右侧预览/摘要区`：
- 左侧主区：当前被编辑的实体
- 右侧辅区：即时预览、运行摘要、保存反馈

主区顶部使用分段切换，而不是长页面堆叠：
- `角色`
- `世界书`
- `规则`

单次默认只展开一个类别。

#### Review 的默认交互
- 打开后直接落在一个明确的默认类别
- 用户在当前类别中逐条查看现有实体
- 编辑后点击唯一主按钮 `保存并刷新预览`
- 预览与项目数据即时同步

#### Review 中必须保留的信息
- 当前实体的关键字段
- 当前项目的 preview 结果
- 当前保存状态
- 当前类别中的数量信息

#### Review 中必须弱化的信息
- 大量装饰性卡片边框
- 多列并排的复杂表单阵列
- 与当前编辑无关的其它类别内容

## 舞台骨架
舞台骨架只用于 `reader`。

它吸收参考图的关键优点，但不直接复制：
- 阅读区域保持窄而聚焦
- 背景图像只承担氛围，不承担信息
- 文字直接覆盖在舞台层上
- 选择作为底部命运面板出现

Reader 的核心层级必须是：
1. 当前场景标题、正文、对话
2. 当前可选行动或自由输入
3. 当前规则命中摘要
4. lore / state 等结构化辅助信息

## Reader 桌面端
桌面端采用 `中心主舞台 + 轻量侧信息`。

### 布局
- 中央为固定宽度阅读舞台
- 左侧为世界上下文窄栏
- 右侧为实时状态窄栏
- 两侧栏宽度明显小于中央舞台

### 桌面端中央舞台
中央舞台包含：
- 顶部轻量导航和会话工具
- 当前场景标题
- 场景正文与对话
- 贴近正文的规则命中摘要
- 底部选择面板
- 自由输入面板

正文与选择必须共享最大视觉面积。

### 桌面端侧栏
左侧世界栏只展示：
- 当前激活 lore
- lore 槽位
- 触发原因
- 生命周期摘要

右侧状态栏只展示：
- `event_flags`
- `possibility_flags`
- 角色运行时状态摘要
- 最近规则命中

桌面端侧栏规则：
- 默认可见，但视觉上弱化
- 单条信息尽量短
- 不出现高密度段落
- 不使用与正文同等级的大标题和重背景

## Reader 移动端
移动端采用 `全聚焦单舞台 + 抽屉式辅助信息`。

### 布局
- 全屏只有一个纵向阅读舞台
- 正文覆盖在背景氛围层之上
- 选择面板固定在底部或贴近底部弹起
- lore / state 改为左右抽屉或轻按钮触发的覆盖层

### 移动端保留参考图的关键体验
- 视觉中心集中在中间窄舞台
- 用户默认只处理“继续读”和“做选择”
- 结构化信息不默认摊开

### 移动端辅助信息展开方式
- 左侧抽屉：世界与 lore
- 右侧抽屉：状态与规则
- 展开后覆盖在舞台上层，关闭后回到纯阅读

### 移动端设计约束
- 不保留常驻三栏
- 不在首屏展示大量状态文本
- 抽屉入口必须清晰可发现
- 展开层内容必须支持快速关闭

## Reader 的共通交互
无论桌面端还是移动端，都要共享相同的内容优先级与行为逻辑。

### 规则命中
- 规则命中摘要放在主舞台附近
- 呈现为一到两条短摘要或胶囊，而不是大块警示面板
- 更详细的规则信息留给状态区或抽屉

### 选择
- 选择显示为统一的底部决策区
- 每个选项只展示标题和必要摘要
- 不默认暴露过长的系统解释
- 被规则锁定的选项要明显不可用，但解释保持简短

### 自由输入
- 自由输入仍属于主舞台的一部分
- 不放进侧栏
- 在没有输入需求时可折叠或次级显示

### 回溯与会话工具
- 属于会话控制，不应挤占正文区域
- 在桌面端放入顶部轻量工具区
- 在移动端收纳进顶部图标或更多菜单

## 视觉语言
整体视觉方向定义为：
- 安静
- 清楚
- 轻戏剧性
- 低压力

### 工作台视觉
- 更浅、更干净、更像编辑器
- 组件边界清晰，但层次克制
- 主按钮明确，次要操作退后

### 舞台视觉
- 更深、更聚焦、更像阅读器
- 背景允许有图像氛围层
- 文本对比度必须足够高
- 装饰不能影响长文本阅读

### 两者共享的统一元素
- 同一品牌名与阶段条
- 同一按钮语义
- 同一圆角和间距系统
- 同一状态颜色语义

## 组件边界
为避免页面再次长回单个大组件，界面层应按职责拆分：

### 工作台侧
- 阶段容器
- Review 类别切换器
- 当前实体编辑面板
- 即时预览面板
- 保存反馈条

### 舞台侧
- Reader 主舞台
- Reader 顶部工具条
- 选择面板
- 自由输入面板
- 世界侧栏 / 世界抽屉
- 状态侧栏 / 状态抽屉

同一份运行时数据通过不同布局消费，不新增 reader 专属后端契约。

## 状态与数据流要求
本次重构不修改已有后端接口，但界面消费方式要明确：

### Review
- `get_project` 提供当前可编辑真源
- preview 接口驱动右侧即时预览
- 保存后立即刷新 project 与 preview

### Reader
- `ScenePayload` 仍是唯一主输入
- 桌面端与移动端都消费：
  - `scene`
  - `active_lore`
  - `active_rules`
  - `story_state`
  - `session`

区别只在展示，不在数据契约。

## 错误处理
### Review
- 保存失败时，在主编辑区附近给出明确反馈
- preview 失败不应清空当前编辑内容
- 用户需要知道是“保存失败”还是“预览失败”

### Reader
- 推进剧情失败时，错误提示必须贴近主舞台
- 侧栏或抽屉错误不应阻断正文显示
- 抽屉加载失败时允许关闭并继续阅读

## 验收标准
### Review
- 用户进入审阅后，首屏能立刻判断自己正在编辑哪一类实体
- 页面默认不同时堆出角色、世界书、规则三大块长表单
- 保存操作只有一个明确主按钮
- 保存后预览和后续 `start_session` 反映最新值

### Reader 桌面端
- 首屏视觉焦点明确落在正文和选择
- 左右信息栏存在，但明显弱于中央舞台
- 规则命中不再占用整块高噪声区域

### Reader 移动端
- 首屏没有常驻侧栏
- 用户可以只靠正文和底部选择顺畅游玩
- 世界与状态信息能通过抽屉快速查看和关闭

### 一致性
- 从 review 进入 reader 时，用户能感知到任务切换，但仍知道自己还在同一个产品里
- 桌面端与移动端共享相同的数据含义和交互逻辑

## 实施边界
本设计只覆盖本仓库当前的单页流转：
- `import`
- `building`
- `review`
- `reader`
- `ending`

其中本次重构重点为：
- 首页阶段容器与整体骨架
- review 布局重组
- reader 桌面端重组
- reader 移动端重组

`ending` 只做必要的风格对齐，不单独扩展新交互。
