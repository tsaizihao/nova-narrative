# 面向普通玩家的长期小说改编操作系统设计

## 文档目标

本文档定义 `叙世者` 从当前 `导入 -> 提炼 -> 编译 -> 审阅 -> 游玩` 原型，演进为长期、通用、可扩展的 `小说改编操作系统` 的目标形态。

本文档回答以下问题：

- 长期产品到底要成为什么，而不只是“当前原型还能加什么”
- 为什么长期方向应选择“改编操作系统”而不是模板工厂或全动态故事机
- 这套系统如何面向普通玩家提供消费级的一键改编体验
- 这套系统的核心中间层、数据对象、运行时边界和阶段路线应该如何设计
- 现有仓库中哪些能力可以保留，哪些必须重做或升级

本文档不是实现计划，也不是一次性想法记录。它是后续架构、计划和实现分解的长期设计依据。

## 设计结论

长期方向选择：

- 底层按 `改编操作系统` 建设
- 上层吸收 `向导驱动模板工厂` 的消费级首屏体验
- 运行时保留 `局部动态演化` 的能力，但不走全动态失控路线

一句话定义：

`叙世者` 的长期目标，不是做一个小说编辑器，也不是做一个 AI 即兴讲故事器，而是做一个面向普通玩家的小说改编操作系统：先把任意原著转换成可改编、可约束、可重组的故事中间层，再通过问答、提案和正式生成流水线，把它产出为可直接游玩的互动游戏。

## 已确认的产品前提

本设计建立在以下前提上：

- 第一目标用户是 `普通玩家`
- 平台要支持 `任意小说导入`
- 改编自由度要高，原著应被视为 `素材库` 而不是只允许小修小补的圣经
- 长期要优先支持的重构能力是：
  - `玩家插入原著`
  - `关系网络重构`
- 人物核心设计原则是：
  - `尽量保留原著人物的核心性格`
  - `关系走向允许重排`
- 产品形态优先级是：
  - `先在本软件内跑通`
  - `导出能力后置`
- 视觉资产策略是：
  - `背景图 + 立绘` 尽早进入长期架构
  - 配音不作为早期核心依赖
- 交互形态是：
  - `选项为主`
  - `少量自由输入辅助`
- 生成体验是：
  - `完整偏好问答`
  - `先给出多种改编提案`
  - `用户选择后再正式生成`
- 游戏生成策略是：
  - `主干稳定`
  - `局部动态演化`
- 质量策略是：
  - `质量优先`
  - 允许更长生成时间，换取更稳定的首次结果
- 平台能力来源是：
  - `云端模型优先`
- 部署形态是：
  - `桌面软件优先`
  - 项目主要保存在本地

## 为什么不是另外两条路线

### 为什么不是“向导驱动模板工厂”

模板工厂路线的优点是产品表面会更快落地，但它会在长期阶段暴露出三个问题：

- 高自由改编会被模板边界卡死
- 任意小说的通用性会明显不足
- 玩家插入原著与关系重构很容易退化成“换皮模板”

模板工厂适合快速验证用户是否愿意点按钮生成游戏，但不适合作为长期底座。

### 为什么不是“全动态故事机”

全动态路线虽然自由度高，但它会直接破坏本项目最重要的两个前提：

- 人物核心性格必须尽量保持
- 游戏必须像“可反复游玩和可验证”的作品，而不是每次都换一套逻辑

如果运行时大部分结构都临时生成，那么：

- 角色一致性最难控制
- 关系重构会失去边界
- 主干剧情很难锁定
- 结局条件难以验证
- 后续导出几乎不可控

因此，动态能力只能作为 `主干编排之后的局部增强`，不能替代正式生成系统。

## 产品定义

`叙世者` 的长期产品定义如下：

- 输入是任意小说原文
- 中间层不是简单摘要，而是一套 `可改编系统`
- 用户不是脚本作者，而是 `改编方向的决策者`
- 系统不是生成一堆散文，而是编排 `可游玩的互动游戏`
- 游玩体验是：
  - 有稳定主线
  - 有关键分支
  - 有多角色关系线
  - 有结局体系
  - 同时保留局部动态内容

更具体地说，这套产品既不是：

- 传统 VN 编辑器
- 电子书阅读器
- 角色扮演聊天工具
- 纯 prompt 编排器

它是一套把原著变成“可重写、可约束、可计算、可游玩”的 `改编操作系统`。

## 长期主流程

长期用户主流程应固定为：

1. 导入原著
2. 原著理解与改编内核构建
3. 偏好问答
4. 多提案生成
5. 用户选择提案
6. 正式生成可玩包
7. 自动补完与轻审阅
8. 在软件内游玩

其中最重要的产品原则是：

`用户做选择，系统做编排`

这意味着普通玩家在前台看到的应该只有：

- 导入
- 问答
- 提案对比
- 一键生成
- 试玩与少量确认

而系统在后台完成：

- 原著理解
- 人物核心性格锚定
- 关系重构可行性分析
- 提案生成与排序
- 正式剧情编排
- 运行时动态边界控制
- 资产需求规划

## 总体架构

长期平台的架构应分为九层能力，其中前四层构成长期底座。

### 1. 原著导入层

职责：

- 导入任意小说文本
- 切卷、分章、分段
- 保留原文结构和基础元数据
- 为后续理解提供稳定输入

输出：

- `SourceNovel`

### 2. 原著理解层

职责：

- 抽取人物、地点、事件、阵营、时间线
- 识别原著关系图
- 抽取世界规则和叙事母题
- 提炼人物核心性格锚点
- 标记原著中“可改”和“不可轻易改”的约束

输出：

- `CanonCharacter`
- `CanonWorldModel`
- 初步 `AdaptationConstraintSet`

### 3. 改编约束层

职责：

- 保存人物核心性格锚点
- 保存关系重构边界
- 保存不可被运行时篡改的主干规则
- 保存原著世界与变体设定之间的映射约束

这一层的作用不是限制改编自由，而是防止改编失控。

### 4. 改编操作系统内核

这是长期平台最核心的中间层，由以下部分组成：

- 角色本体库
- 关系图谱
- 事件图谱
- 世界规则集
- 可变剧情槽位
- 改编约束集

关键要求：

- 所有提案都必须从该内核中产生
- 所有正式生成都必须基于该内核编排
- 所有运行时动态变化都必须受该内核约束

### 5. 向导问答层

职责：

- 用适合普通玩家的问答方式收集偏好
- 把答案转成结构化改编意图

问答对象至少应覆盖：

- 代入方式
- 主角模式
- 关系重构方向
- 游戏篇幅
- 风格倾向
- 视觉偏好
- 结局倾向

### 6. 提案生成层

职责：

- 根据 `改编内核 + 用户意图` 生成多个结构化改编提案
- 给普通玩家明确展示“这会变成什么样的故事”

### 7. 正式生成层

职责：

- 把选中的提案编成稳定主干和关键分支
- 生成可游玩场景束
- 生成结局网
- 生成角色关系线
- 生成动态槽位
- 生成资产需求清单

### 8. 运行时演化层

职责：

- 结算选择
- 接收少量自由输入
- 在场景契约范围内生成局部动态内容
- 做一致性校验
- 把合法变化落回运行时状态

### 9. 资产与表现层

职责：

- 背景图规划与绑定
- 立绘规划与绑定
- 章节表现资源管理
- 视觉风格与提案风格对齐

## 提案系统与正式生成系统

长期平台必须明确把“提案生成”和“正式生成”拆成两个系统。

### 提案系统

职责：

- 给用户几个方向明确、互相区分的改编方案

输入：

- `AdaptationKernel`
- `PlayerIntentProfile`
- `AdaptationConstraintSet`

输出：

- `ProposalBlueprint[]`

每个提案必须包含：

- 主角模式
- 关系变化摘要
- 主线目标
- 主要冲突
- 推荐篇幅
- 视觉气质
- 风险提示

提案不是一段营销文案，而必须是后续正式生成可消费的结构化蓝图。

### 正式生成系统

职责：

- 把用户选中的提案编成真正可玩的故事包

输入：

- `ApprovedAdaptationPlan`
- `AdaptationKernel`
- `AssetStrategy`

输出：

- `StoryGamePackage`

正式生成内部建议继续拆成五个子系统：

1. `主干编排器`
   - 决定章节结构、主要目标、关键转折、结局布局
2. `角色关系编排器`
   - 分配多角色关系线的推进节奏
   - 保证人物核心性格不崩
3. `场景束生成器`
   - 把剧情拆成可游玩的场景组，而不是零散文本
4. `动态槽位注入器`
   - 标记运行时允许动态变化的位置
5. `资产规划器`
   - 产出背景图、立绘、状态变体和章节表现需求

关键原则：

- 提案负责选方向
- 正式生成负责做成品
- 两者不能混在一次大 prompt 中完成

## 运行时与动态演化边界

长期平台必须采用：

`主干稳定，局部动态`

### 允许运行时动态变化的部分

- 局部对话措辞和情绪表现
- 小型支线事件
- 少量自由输入后的即时回应
- 背景说明和关系解释的补充文本
- 不改变结论的场景过渡与演出细节

### 禁止运行时擅自改动的部分

- 已锁定的主线章节顺序
- 已锁定的关键转折
- 已建立的主要关系走向
- 已锁定的结局条件
- 人物核心性格锚点
- 世界规则
- 已锁定事实

### 建议的运行时分层

1. `主干状态层`
   - 保存章节、关键选择、关系阶段、结局进度和锁定事实
2. `场景契约层`
   - 规定当前场景允许出现的角色、目标、冲突、情绪与动态槽位
3. `动态演化层`
   - 在契约约束下生成局部内容
4. `一致性裁判层`
   - 检查是否违反人物锚点、世界规则和已锁定事实
5. `结果落账层`
   - 只把被允许的变化写回运行时状态

一句话原则：

`动态生成只能填空，不能改卷子`

## 数据模型与关键对象

长期平台必须把偏好、提案、计划、游戏包和运行时显式分层。

### 1. SourceNovel

表示原始输入小说。

职责：

- 保存原文
- 保存章节结构
- 保存基础元数据

### 2. CanonCharacter

表示原著人物本体。

关键字段建议包括：

- 基础身份
- 原著作用
- 动机
- 核心性格锚点
- 禁改边界
- 原著关键关系

### 3. CanonWorldModel

表示原著世界模型。

关键字段建议包括：

- 地点
- 时间线
- 事件
- 阵营
- 世界规则
- 原著事实

### 4. AdaptationKernel

表示长期平台的统一改编内核。

组成部分包括：

- `CharacterIdentityGraph`
- `RelationshipGraph`
- `EventGraph`
- `WorldRuleSet`
- `AdaptationConstraintSet`
- `MutableNarrativeSlots`

### 5. PlayerIntentProfile

表示用户问答结果的结构化对象。

它不能只是备注文本，而要能被提案系统和正式生成系统直接消费。

### 6. ProposalBlueprint

表示单个改编提案。

建议包含：

- 提案标题
- 主角模式
- 关系重构摘要
- 主线目标
- 风格标签
- 篇幅建议
- 风险和不确定性

### 7. ApprovedAdaptationPlan

表示用户选中的正式改编计划。

它是正式生成的唯一入口，不应直接从 `PlayerIntentProfile` 跳到 `StoryGamePackage`。

### 8. StoryGamePackage

表示正式生成后的可玩包。

建议包含：

- 主干章节树
- 关键分支图
- 场景束
- 关系线推进表
- 结局网络
- 动态槽位清单
- 资产绑定清单

### 9. RuntimeSession

表示一次游玩。

建议保存：

- 当前章节位置
- 已做关键选择
- 当前关系状态
- 已锁定事实
- 已触发动态内容
- 当前可见目标

### 10. AssetManifest

表示背景图、立绘和表现资源清单。

建议保存：

- 场景到背景图的绑定
- 角色到立绘集合的绑定
- 情绪/状态到立绘变体的绑定
- 提案风格到表现资源风格的绑定

## 现有仓库与长期目标的映射

当前仓库已经具备以下可保留基础：

- 导入与章节切分
- 角色 / 世界书 / 规则的显式建模
- 审阅工作台
- reader runtime shell
- session 生命周期

这些能力说明当前仓库已经具备“原著 -> 世界模型 -> 游玩”的雏形，但长期目标与现状之间仍有明显差距。

### 可以保留并扩展的部分

- 当前导入入口
- 项目 / session 持久化主线
- review workspace 的“轻审阅”形态
- runtime snapshot 和 reader 外壳
- worldbook / rule / story state 的显式语义

### 必须重做或大幅升级的部分

- 外部模型分析策略
  - 不能再按单次全文 prompt 处理长期目标
- compile 逻辑
  - 不能停留在少量样板场景和少量固定结局
- 用户前台产品流
  - 需要偏好问答和提案系统
- runtime 动态能力
  - 需要场景契约层和一致性裁判层

## 分阶段落地路线

推荐把长期平台拆成六个阶段推进。

### 阶段 1：原著理解升级

目标：

- 把当前提炼层升级为长期可复用的原著理解层

工作：

- 多模型、多阶段抽取
- 人物核心性格锚点抽取
- 关系图谱抽取
- 事件图谱抽取
- 原著约束抽取

完成标准：

- 不再只得到角色卡、规则草案和世界书条目
- 能稳定产出长期内核所需的基础对象

### 阶段 2：改编操作系统内核

目标：

- 建立 `AdaptationKernel`

工作：

- 新增长期对象层
- 把“原著理解”和“游戏生成”解耦
- 让后续模块全部围绕统一中间层工作

完成标准：

- 提案系统、正式生成系统和 runtime 都能以同一内核为输入

### 阶段 3：偏好问答与多提案系统

目标：

- 让普通玩家第一次真正感知“改编产品”而不是“编辑器”

工作：

- 构建问答向导
- 构建 `PlayerIntentProfile`
- 构建多提案比较页
- 构建提案选择流程

完成标准：

- 用户能完成问答
- 用户能在多个方向明确的提案中做选择

### 阶段 4：正式生成流水线

目标：

- 用稳定编排替代当前样板 compile

工作：

- 主干编排器
- 角色关系编排器
- 场景束生成器
- 动态槽位注入器
- 资产规划器

完成标准：

- 输出是真正的 `StoryGamePackage`
- 不再只是少量样板场景加固定终点

### 阶段 5：局部动态运行时

目标：

- 让游戏变得更活，同时不失控

工作：

- 场景契约层
- 一致性裁判层
- 结果落账层
- 局部动态事件与文本生成

完成标准：

- 运行时动态内容能够增强体验
- 但不会破坏人物底色与主干结构

### 阶段 6：背景图与立绘系统

目标：

- 把“可玩原型”提升为“像游戏的产品”

工作：

- 资产规划与引用
- 提案风格到视觉风格的映射
- 角色状态到立绘变体的映射
- 场景到背景图的映射

完成标准：

- 游戏包可以稳定消费背景图与立绘
- 用户不需要先学资产管理，也能得到可展示的结果

## 推荐实施顺序

推荐先做：

- 阶段 1
- 阶段 2

原因：

- 没有稳定内核，提案系统和正式生成系统都会退化成一次性脚本

推荐第二波做：

- 阶段 3
- 阶段 4

原因：

- 这一波会让普通玩家第一次真正体验到“问答 -> 选提案 -> 生成作品”的核心价值

推荐第三波做：

- 阶段 5
- 阶段 6

原因：

- 这一波负责把产品从“能工作”推进到“更好玩、更像成品”

## 最大风险与对应约束

### 风险 1：高自由改编退化成全动态乱写

后果：

- 人物崩坏
- 关系线漂移
- 主干失控

约束：

- 动态生成只能填空，不能改卷子

### 风险 2：普通玩家一键向导退化成模板工厂

后果：

- 产品短期像样，长期无扩展性
- 任意小说支持会变差

约束：

- 所有提案和正式生成都必须建立在 `AdaptationKernel` 上

### 风险 3：对象边界不清，系统耦合失控

后果：

- 后续换模型、加资产、加导出都很痛苦

约束：

- `PlayerIntentProfile`
- `ProposalBlueprint`
- `ApprovedAdaptationPlan`
- `StoryGamePackage`
- `RuntimeSession`

这五层对象必须显式分层，不能混成一个大对象。

## 验收标准

当长期平台演进到正确方向时，应能满足以下验收结果：

- 用户可以导入任意小说
- 用户可以通过完整偏好问答表达改编方向
- 系统可以产出多个差异明确的改编提案
- 用户可以选择提案并生成稳定可玩的作品
- 系统可以支持玩家插入原著和关系网络重构
- 原著人物核心性格仍然能被约束住
- 游戏主干稳定，局部内容动态
- 背景图和立绘可以被纳入正式资产清单
- 整个流程仍然以桌面软件内直接游玩为第一目标

## 结论

`叙世者` 的长期方向，不应继续沿着“更复杂的当前 compile”线性加功能，也不应退回模板工厂或全动态故事机。

正确方向是：

- 先建设 `改编操作系统`
- 再建设 `普通玩家向导 + 多提案产品层`
- 最后在 `主干稳定` 的前提下，加入 `局部动态演化` 和 `资产表现力`

这条路线实现周期更长，但它是唯一同时满足以下目标的方案：

- 普通玩家可用
- 任意小说通用
- 高自由改编
- 人物不崩
- 长期可扩展
- 能从当前仓库逐步演进而不是推倒重来
