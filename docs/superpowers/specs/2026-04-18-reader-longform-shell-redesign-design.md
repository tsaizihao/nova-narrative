# 暖色长页阅读器重设计

## 背景
当前阅读阶段沿用的是 `左侧世界侧栏 + 中间舞台 + 右侧世界状态` 的三栏结构。它能完整暴露系统信息，但更像一个运行时工作台，而不像用户真正愿意停留的视觉小说阅读器。

这和产品现在的目标不一致。用户已经明确表达两点：

- 阅读页应该更接近“视觉小说 / 互动小说”的体验，而不是审阅工具继续延伸
- 可参考 SillyTavern 这类长页阅读与底部输入布局，但视觉上仍保持本项目现有的暖色纸面气质，不切换到黑底控制台

因此，本次重设计不再围绕“三栏信息效率”优化，而是改为围绕“沉浸式长页阅读”优化。

## 设计结论
最终方向已经确认如下：

- 主布局采用 `暖色 Tavern 流`
- 主阅读区采用 `无气泡小说流`
- 世界信息与状态信息默认隐藏，改为 `抽屉 / 覆层` 按需打开
- 视觉维持当前项目的米白、宣纸、暖灰、暖金、黛青体系，不采用黑底 UI

一句话概括：

`把阅读阶段从三栏工作台改造成一张可连续往下读的互动小说纸页，并把系统信息收纳到抽屉里。`

## 目标

### 体验目标
- 让用户一进入阅读阶段就感知到“这是互动小说 / 视觉小说阅读器”，而不是“调试控制台”
- 让正文成为绝对视觉主角，信息面板退到次级入口
- 让阅读和输入形成一个连续循环：读一段、点继续、看到新段落、再决定是继续还是输入行动
- 保持当前系统的结构化优势，但把它们藏在更自然的入口后面

### 产品目标
- 不改动底层 runtime 数据模型即可完成阅读器壳层升级
- 桌面和移动端共享同一套信息架构，只在呈现方式上做适配
- 为后续加入角色立绘、场景图、章节皮肤、自动播放节奏控制留下空间

## 非目标
- 本次不重做导入页、审阅台、适配内核、故事编译器或存储结构
- 本次不引入黑底赛博聊天界面
- 本次不引入复杂动画、古风贴图或高密度装饰纹样
- 本次不做真正的“左右聊天气泡角色扮演 UI”
- 本次不解决角色立绘生成或场景背景素材生产问题

## 核心设计原则

### 1. 正文优先
页面首先是一张可读的互动小说纸面，其次才是可操作的系统界面。

### 2. 信息收纳
世界书、规则命中、状态、时间线等仍然重要，但不应默认占据首屏横向空间。

### 3. 长页推进
剧情不是一屏一屏切换，而是在当前场景上下文中连续追加，使用户形成“正在读一本会回应我的书”的感受。

### 4. 暖色纸面，不做黑底
本项目的基调应是纸面、小说、戏台、阅读器，而不是聊天工具或终端。

### 5. 运行时能力不丢
当前系统的状态、规则、回溯、自动播放、自由输入等能力必须保留，只改变组织方式和优先级。

## 新阅读器信息架构

新阅读器由四层组成：

1. `顶部轻量标题条`
2. `主阅读流`
3. `底部固定操作坞`
4. `按需展开的信息抽屉`

### 顶部轻量标题条
顶部改成薄而稳定的一条，不再做大面积卡片。

包含：
- 返回审阅台
- 作品名
- 当前章节 / 场景名
- 可选的轻量状态标签，例如自动播放中、当前角色身份

不包含：
- 大块统计卡片
- 常驻世界信息摘要
- 与正文争夺视觉主导权的巨大标题盒子

目标是让用户快速知道“我在读什么”，然后立即进入正文。

### 主阅读流
主阅读区是一块长页内容区，它承担阅读阶段的绝大部分空间。

阅读流中的信息块按时间顺序向下排列，典型结构如下：

1. 章节或场景起始标签
2. 场景摘要 / 开场段
3. 旁白正文
4. 角色发言
5. 关键句高亮
6. 动作结果或系统反馈
7. 下一轮交互入口

关键点：
- 不采用左右聊天气泡
- 不让每个段落都成为独立厚重卡片
- 保持“小说正在展开”的连续感

### 底部固定操作坞
底部固定操作坞始终存在，不跟随正文滚走。

它承担三类交互：

- 主推进：`继续`
- 辅助控制：`自动播放`、`重试`、`清除输入`
- 用户输入：自由输入框 / 行动描述框

用户在任何滚动位置，都能立即触达下一步交互。

### 信息抽屉
常驻侧栏取消，改成按需打开的抽屉或覆层。

左抽屉：
- 世界信息
- 人物
- lore
- 时间线
- 关系与设定

右抽屉：
- 世界状态
- 规则命中
- 事件旗标
- 可能性变化
- 最近动作日志
- 回溯节点

抽屉默认关闭，不压缩主阅读区宽度。

## 主阅读流排版规则

### 整体样式
主阅读流采用 `无气泡小说流`。

也就是说：
- 正文默认就是小说正文
- 角色发言并入正文流
- 不把每句台词包装成聊天消息框

阅读区应该更接近：
- 长篇互动小说
- 带输入能力的阅读器
- 暖色化的 SillyTavern 长页模式

而不是：
- IM 聊天窗
- 分栏控制台
- 视觉上很重的气泡对话器

### 旁白
旁白是默认正文样式：
- 最大阅读占比
- 最高可读性
- 足够舒展的行高
- 作为整页排版的基底

### 角色发言
角色发言仍然在正文中连续呈现，但比旁白多一层轻量识别：
- 段前显示角色名
- 角色名可采用暖金或黛青色小标题
- 发言正文仍使用连续段落，而不是聊天气泡

### 关键句和规则触发
关键句、高风险提示、规则命中解释可以用局部强调，但强调必须克制。

允许方式：
- 关键词暖金强调
- 单句加粗
- 段前标签
- 极轻背景带

禁止方式：
- 大面积高饱和底色
- 高频弹窗打断阅读
- 多种强调色同时竞争

### 场景块
阅读流按“场景块”组织，而不是按“UI 模块卡片”组织。

每个场景块可能包含：
- 场景名
- 场景起始段
- 叙述与发言
- 一次关键动作
- 该动作的结果

继续推进时，新的场景块追加到旧内容下方。

## 底部固定操作坞设计

### 结构
操作坞拆成三部分：

1. `主动作区`
2. `辅助控制区`
3. `输入区`

### 主动作区
默认主动作是 `继续`。

当当前节点存在明确分支时，主动作区切换成 2 到 4 个选择按钮。

规则：
- `继续` 永远是最醒目的主按钮
- 如果当前是选项节点，选项按钮取代单一继续按钮
- 主动作区优先级高于其他控制

### 辅助控制区
放置次级动作：
- 自动播放
- 重试
- 清除输入

原则：
- 不抢主推进的视觉权重
- 可见但不喧宾夺主

### 输入区
输入区长期存在，不因当前是否有选项而完全消失。

文案方向：
- “以某角色身份发言，描述你的行动或接下来发生的事”

这样可以保留傻瓜酒馆式“随时接话”的心理预期。

### 继续与追加
点击继续后的行为不是切页，也不是重载整个场景卡片，而是：

1. 运行时完成下一步推进
2. 新叙事块追加到长页末尾
3. 视口平滑滚到新内容附近

用户因此获得连续阅读感，而不是页面跳变感。

### 自动播放
自动播放只负责连续推进可直接阅读的段落。

遇到以下情况必须暂停：
- 分支选项
- 自由输入要求
- 错误
- 关键剧情停顿
- 抽屉展开

### 重试
`重试` 不是“重新开始整个场景”，而是“重做上一轮失败或异常生成”。

因此它：
- 默认不做主按钮
- 仅在失败场景下高亮
- 语义明确指向“上一轮动作”

## 信息抽屉设计

### 左抽屉：世界信息
左抽屉承载较稳定、阅读时偶尔查询的信息：
- lore
- 人物卡
- 时间线
- 关系
- 世界设定

它的体验像“翻出设定附录”。

### 右抽屉：状态 / 日志
右抽屉承载更临时、更运行时的信息：
- 当前规则命中
- 事件旗标
- 可能性变化
- 最近动作结果
- 回溯节点

它的体验像“查看当前局面和日志”。

### 抽屉行为
- 默认关闭
- 打开后覆盖在阅读页上层
- 不重新排布主阅读区
- 保持底部操作坞可见
- 打开时自动播放暂停

### 内容复用策略
现有 `StoryCodexPanel` 和 `StoryStatePanel` 中的内容逻辑不废弃，而是拆成抽屉内容模块复用。

这能降低重写成本，并保持信息内容的一致性。

## 视觉语言

### 色彩
维持现有暖色体系：
- 米白
- 宣纸色
- 暖灰
- 暖金
- 黛青 / 青绿

不采用黑底方案。

### 层级
- 标题使用更具文学感的衬线气质
- 正文使用高可读性中文排版
- 强调色只保留暖金与黛青两条主线

### 容器感
目标是削弱“到处都是同等级卡片”的工作台感。

做法：
- 减少并列大面板
- 减轻边框重量
- 减轻阴影
- 让正文容器更像一张连续纸面，而不是一组工具盒

### 装饰
允许：
- 轻纸纹
- 极轻渐变
- 少量暖金分隔
- 克制的抽屉滑出动效

避免：
- 大量古风花边
- 高密度纹样
- 贴图感很重的戏曲装饰

目标是“有小说气质”，不是“古风模板页”。

## 组件演进方案

### 保留的能力边界
本次只改阅读器壳层，不改：
- runtime session 数据结构
- 场景推进逻辑
- 规则判断逻辑
- 回溯逻辑
- 自由输入逻辑
- 审阅台与导入台流程

### 新的桌面壳层
`RuntimeStageShell` 仍然是阅读阶段入口，但桌面端结构不再渲染三栏常驻布局。

新的桌面阅读壳层会变成：
- 顶部标题条
- 主阅读流
- 底部固定操作坞
- 左右抽屉

### ReaderStage 的职责变化
当前 `ReaderStage` 更像中心舞台卡片。

重设计后，它应转为：
- 长页叙事流渲染器
- 负责正文块、角色发言、关键句、动作结果、选项块的连续渲染

### 抽屉内容模块
现有：
- `StoryCodexPanel`
- `StoryStatePanel`

后续拆分方向：
- 面板头部容器逻辑与内容列表逻辑分离
- 内容项可被桌面抽屉与移动覆层共同使用

### 移动端
移动端不维持另一套完全不同的信息架构。

共享原则：
- 主阅读流保持一致
- 信息通过底部抽屉或全屏覆层进入
- 底部操作坞仍保留核心交互

## 状态与交互规则

### 加载态
加载时不出现三栏骨架占位，而是更像“阅读器正在打开这一幕”：
- 作品名
- 当前阶段说明
- 轻量骨架屏

### 错误态
错误态保留在阅读器上下文中，而不是跳回工具感很强的错误卡片。

应包含：
- 错误提示
- 重试按钮
- 返回审阅台入口

### 结束态
结束态仍然可以保留独立处理，但视觉语言应与新阅读器一致：
- 纸面感
- 阅读完成感
- 后续动作清晰

## 可访问性
- 正文行高需要优先服务长时间阅读
- 暖金强调色必须满足可读性，不能只追求气氛
- 抽屉必须支持键盘关闭与焦点返回
- 自动播放必须有明确开关状态和可随时中断能力
- 输入区不能因为滚动或抽屉展开而失去可聚焦性

## 测试策略

### 组件测试
新增或改造后的测试应覆盖：
- 长页阅读流正确渲染当前场景
- 主推进区与分支选择区的切换
- 输入区始终存在
- 抽屉开关行为
- 抽屉打开时自动播放暂停
- 错误态下重试按钮显现

### 页面流测试
页面级测试应覆盖：
- 从审阅台进入阅读阶段后加载新壳层
- 继续推进时内容追加而不是切页替换
- 选项节点主按钮区正确切换
- 返回审阅台仍然可用

### 回归重点
尤其关注以下回归风险：
- 原有 runtime 快照结构仍能驱动新 UI
- 自动播放在特殊节点的暂停规则
- 回溯入口从常驻侧栏迁移到抽屉后是否仍可用
- 移动端覆层与桌面端抽屉是否共享同一内容源

## 风险与取舍

### 风险 1：信息太深，用户找不到
把状态和 lore 全部藏进抽屉后，可能出现“沉浸更强，但信息透明度下降”。

缓解方式：
- 给抽屉入口明确名称
- 在正文中保留极轻的规则命中 / 状态提示
- 对首次使用用户给出一次性引导

### 风险 2：长页追加后变得过长
长页阅读天然会越读越长。

缓解方式：
- 以“场景块”为最小结构
- 保留回到最新内容的入口
- 后续可考虑“本幕折叠”能力，但不在本次范围内

### 风险 3：像 Tavern，但失去本产品的小说气质
如果直接照搬聊天流，很容易变成角色聊天工具而不是互动小说。

缓解方式：
- 保留章节/场景标签
- 使用连续正文排版而非气泡
- 使用暖色纸面和文学化标题层级

## 推荐落地顺序

1. 重构桌面阅读壳层骨架
2. 将左右常驻面板改造成抽屉容器
3. 将主舞台改造成长页叙事流
4. 落地底部固定操作坞
5. 调整自动播放、重试、输入区状态规则
6. 做移动端对应覆层方案
7. 统一视觉语言并补齐回归测试

## 最终结论
这次重设计不是“把当前阅读页美化一下”，而是一次明确的信息架构切换：

- 从 `三栏工作台`
- 变成 `暖色长页互动小说阅读器`

主阅读流成为主角，系统信息退居抽屉；输入与推进固定在底部；整体体验靠近 SillyTavern 的长页沉浸感，但保持本项目自己的暖色纸面和互动小说气质。
